# Publish — Agent Guide

You are preparing a new npm release of anchor-click. Follow every step in order. Do not skip steps. Stop and ask the user before any destructive or irreversible action (push, publish) unless they have explicitly told you to proceed autonomously.

---

## Step 1 — Verify preconditions

Run the following checks before doing anything else. Stop if any check fails and report the problem to the user.

```bash
# Must be on the main branch
git branch --show-current

# Working tree must be clean (no uncommitted changes)
git status --porcelain

# npm must be logged in
npm whoami
```

---

## Step 2 — Determine the release type

Read `git log` since the last tag to understand what changed:

```bash
git log $(git describe --tags --abbrev=0)..HEAD --oneline
```

If there is no previous tag:

```bash
git log --oneline
```

Then propose a version bump to the user based on these semver rules:

| Type | When | Example |
|------|------|---------|
| **patch** | Bug fixes, documentation updates, internal refactoring with no API changes | `1.0.0` → `1.0.1` |
| **minor** | New options, new methods, new behaviour — fully backwards compatible | `1.0.0` → `1.1.0` |
| **major** | Breaking changes to the public API (renamed/removed options, changed return shape) | `1.0.0` → `2.0.0` |

Present your proposed version bump and reasoning to the user. Wait for confirmation before continuing.

---

## Step 3 — Propose changelog entries

Based on the git log from Step 2, draft changelog entries grouped under the appropriate headings:

- **Added** — new features or options
- **Changed** — changes to existing behaviour (non-breaking)
- **Fixed** — bug fixes
- **Removed** — removed features (typically major)
- **Deprecated** — features that will be removed in a future release

Present the draft to the user. Wait for confirmation or edits before writing anything to disk.

---

## Step 4 — Update `package.json`

Update the `version` field to the confirmed new version. Do not change anything else.

---

## Step 5 — Update `CHANGELOG.md`

Prepend a new section above the previous release. Use today's date in `YYYY-MM-DD` format.

```markdown
## [X.Y.Z] - YYYY-MM-DD

### Added
- ...

### Fixed
- ...
```

Only include headings that have entries. Do not include empty sections.

---

## Step 6 — Build

```bash
npm run build
```

Verify the command exits with code 0. If it fails, stop and report the error.

---

## Step 7 — Lint

```bash
npm run lint
```

Verify the command exits with code 0. If it fails, stop and report the error.

---

## Step 8 — Test

```bash
npm test
```

Verify all tests pass. If any test fails, stop and report the error. Do not proceed to publish with a failing test suite.

---

## Step 9 — Commit

Stage only the files that should be part of the release commit:

```bash
git add package.json CHANGELOG.md
```

Commit with a conventional commit message:

```
chore: release vX.Y.Z
```

---

## Step 10 — Tag

Create an annotated tag:

```bash
git tag -a vX.Y.Z -m "vX.Y.Z"
```

---

## Step 11 — Confirm before pushing and publishing

Show the user what is about to happen:

- The commit and tag that will be pushed to `origin/main`
- That `npm publish` will be run immediately after

**Wait for explicit confirmation before continuing.**

---

## Step 12 — Push

```bash
git push origin main --follow-tags
```

---

## Step 13 — Publish

```bash
npm publish --access public
```

Verify the command exits with code 0 and confirm the published version to the user.

---

## Notes

- Never edit files in `dist/` directly — they are generated by `npm run build`.
- The `src/anchorClick.d.ts` types are copied to `dist/` by the build script. If the public API changed in this release, verify that the types were updated in `src/anchorClick.d.ts` before building.
- If `npm publish` fails due to authentication, run `npm login` and try again.
